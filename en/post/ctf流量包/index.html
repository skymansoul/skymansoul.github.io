CTF Traffic Capture: Extracting the Flag from PCAP

In CTF traffic challenges, you are usually given a .pcap, .pcapng, or a compressed archive, and asked to recover a flag from network communication records.

These problems are usually classified under Misc, Forensics, or Traffic Analysis. The core skill is not memorizing commands, but reconstructing what happened on the network.

In short, traffic challenges ask:

Who talked to whom? Which protocol was used? What data was sent? Was it encoded, compressed, split, or hidden?

As long as you follow this logic step by step, you can solve most beginner-to-intermediate traffic tasks.

1. Common Tools

1.1 Wireshark

Wireshark is the most common GUI tool for traffic challenges.

It lets you inspect each packet and filter by protocol, IP, port, or keywords.

Common features:

  • Protocol statistics
  • IP conversations
  • Filters for HTTP, DNS, FTP, ICMP, etc.
  • Follow TCP stream
  • Export HTTP transferred files
  • Inspect plaintext in packets

After opening a capture, two key entries are:

Statistics -> Protocol Hierarchy
Statistics -> Conversations

The first shows which protocols exist; the second shows which hosts communicated.

1.2 tshark

tshark is the CLI version of Wireshark, great for batch extraction.

Show basic capture content:

tshark -r traffic.pcap

Show HTTP requests:

tshark -r traffic.pcap -Y "http.request"

Extract DNS query names:

tshark -r traffic.pcap -Y "dns.qry.name" -T fields -e dns.qry.name

Extract ICMP data field:

tshark -r traffic.pcap -Y "icmp" -T fields -e data

1.3 strings

strings is useful for quick plaintext keyword scanning.

strings traffic.pcap | grep -i flag
strings traffic.pcap | grep -i password
strings traffic.pcap | grep -i admin
strings traffic.pcap | grep -i upload

For easy tasks, strings may directly find the flag. If not, it can still reveal suspicious requests, credentials, or filenames.

1.4 binwalk and foremost

If file content was transferred in the capture (images, archives, docs), try file carving:

binwalk traffic.pcap

foremost traffic.pcap

These are supporting tools. In traffic analysis, protocol and conversation analysis usually come first.

2. Basic Workflow After Getting a Capture

2.1 Read the challenge description first

The description usually gives direction, for example:

  • Password leaked during admin login
  • Hacker uploaded a file
  • Secret hidden in DNS queries
  • Flag sent through ICMP
  • Recover keyboard input
  • Find what attacker downloaded

If it mentions HTTP, prioritize HTTP requests and responses. If it mentions DNS, prioritize domain extraction. If it mentions keyboard, consider USB HID traffic. If it mentions upload/download, pay attention to exported files.

Often the description already tells you the key protocol.

2.2 Check protocol statistics

In Wireshark:

Statistics -> Protocol Hierarchy

Focus on:

  • Plaintext protocols such as HTTP, FTP, DNS, ICMP
  • Special traffic such as USB, Bluetooth, 802.11
  • TCP/UDP ratio anomalies
  • Presence of encrypted TLS traffic

Examples:

  • Heavy HTTP proportion: likely web requests/responses matter
  • Many strange long DNS names: possible DNS tunneling
  • Many ICMP packets: inspect data
  • FTP present: look for plaintext credentials and transferred files

2.3 Check conversations

Open:

Statistics -> Conversations

Focus on:

  • Which IP pair has the most traffic
  • External suspicious IPs
  • Large data-volume connections
  • Suspicious ports
  • Obvious upload/download behavior

Common ports to remember:

21    FTP
22    SSH
25    SMTP
53    DNS
80    HTTP
110   POP3
143   IMAP
443   HTTPS
3306  MySQL
6379  Redis

If one TCP conversation is much larger than others, it may contain file transfer or critical data.

3. Common Wireshark Filters

3.1 Filter by protocol 

http
dns
ftp
icmp
tcp
udp
usb

3.2 Filter by IP 

ip.addr == 192.168.1.10
ip.src == 192.168.1.10
ip.dst == 192.168.1.10

3.3 Filter by port 

tcp.port == 80
tcp.srcport == 8080
tcp.dstport == 21
udp.port == 53

3.4 Search by keyword 

frame contains "flag"
frame contains "password"
frame contains "admin"
frame contains "ctf"

For HTTP:

http contains "flag"
http contains "password"

3.5 Inspect one TCP stream 

tcp.stream eq 0
tcp.stream eq 1
tcp.stream eq 2

Right-click a TCP packet:

Follow -> TCP Stream

This is one of the most important operations in traffic tasks. A single packet might be incomplete, but stream reassembly shows full request/response content.

4. HTTP Traffic Analysis

HTTP is very common in CTF traffic because it is often plaintext.

4.1 Filter HTTP requests

Only HTTP:

http

Only HTTP requests:

http.request

Only GET requests:

http.request.method == "GET"

Only POST requests:

http.request.method == "POST"

POST is especially important for logins, form submissions, and uploads.

4.2 What to inspect in HTTP

Common clue locations:

  • URL parameters
  • POST body
  • Cookie
  • Authorization header
  • Referer
  • User-Agent
  • Response body
  • Uploaded file content
  • Returned archives/images/scripts

Example:

POST /login.php HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

username=admin&password=YWRtaW4xMjM=

YWRtaW4xMjM= looks like Base64. Decoded value:

admin123

This credential may be the key for archive extraction, panel login, or flag submission.

4.3 Export HTTP objects

If files were transferred via HTTP:

File -> Export Objects -> HTTP

Then check file type:

file suspicious_file

If archive: try extract. If image: consider steganography. If script: inspect source code. If filename/response/comment contains strange strings: attempt decoding.

5. FTP Traffic Analysis

FTP is often friendly for CTF because usernames, passwords, and commands may be plaintext.

Filter FTP:

ftp

Common content:

USER admin
PASS 123456

If file transfer exists, inspect:

ftp-data

You can follow related TCP streams to reconstruct transmitted content.

Typical FTP task goals:

  • Find credentials
  • Find uploaded/downloaded filenames
  • Restore transferred files
  • Continue decryption or hidden-content analysis on extracted files

6. DNS Tunneling Analysis

DNS challenges often split data into subdomains.

Filter DNS:

dns

Extract query names:

tshark -r traffic.pcap -Y "dns.qry.name" -T fields -e dns.qry.name

You may see:

ZmxhZ3s.example.com
ZG5zXw.example.com
c2VjcmV0fQ.example.com

Take and concatenate the left parts:

ZmxhZ3sZG5zXwc2VjcmV0fQ

Then try Base64, Hex, URL decode, etc.

Common DNS tunneling signs:

  • Very long subdomains
  • Random-looking domain labels
  • Fragments that can be concatenated across many requests
  • Clear Base64-like character sets
  • Abnormally high query frequency

If the domain contains -, _, digits, mixed case, consider Base64 URL-safe variants.

7. ICMP Traffic Analysis

ICMP is the protocol used by ping. In CTF, data is often hidden in ICMP data fields.

Filter ICMP:

icmp

Extract ICMP data:

tshark -r traffic.pcap -Y "icmp" -T fields -e data

If you get hex like:

666c61677b69636d705f7365637265747d

Decode with Python:

data = "666c61677b69636d705f7365637265747d"
print(bytes.fromhex(data).decode())

Output:

flag{icmp_secret}

Remember deduplication. Ping usually has request and reply; concatenating all ICMP payloads may duplicate data.

Only requests:

icmp.type == 8

Only replies:

icmp.type == 0

8. TCP Stream Following

TCP stream following is a core operation in traffic challenges.

In Wireshark, select a TCP packet and right-click:

Follow -> TCP Stream

This shows full client-server communication for that connection.

Switch streams with:

tcp.stream eq 0
tcp.stream eq 1
tcp.stream eq 2

Prioritize stream-following when:

  • HTTP content appears incomplete
  • FTP transfers files
  • Suspicious port has large data
  • You only see a partial flag or encoded fragment
  • Data is split into many packets

After opening a stream, try display formats:

ASCII
Raw
Hex Dump

If needed, choose Raw and save original data.

9. USB Keyboard Traffic Analysis

Some challenges are not network captures but USB captures. A common task is reconstructing keyboard input.

Filter USB data:

usb.capdata

Extract field:

tshark -r usb.pcap -T fields -e usb.capdata

Keyboard HID data often looks like:

00:00:04:00:00:00:00:00
00:00:05:00:00:00:00:00
00:00:06:00:00:00:00:00

The third byte usually indicates key code:

04 -> a
05 -> b
06 -> c

If first byte is 02 or 20, Shift is typically pressed, so convert to uppercase/symbols.

Typical USB keyboard workflow:

  1. Extract usb.capdata
  2. Remove empty entries
  3. Take key codes
  4. Map using HID keyboard table
  5. Handle special keys: Shift, Backspace, Enter

10. Common Encodings and Decoding Ideas

Flags in captures are not always directly visible. They are often encoded.

10.1 Base64

Features:

  • Usually letters, digits, +, /
  • May end with =
  • Length often multiple of 4

Example:

ZmxhZ3t0ZXN0X2ZsYWd9

Decode:

echo ZmxhZ3t0ZXN0X2ZsYWd9 | base64 -d

10.2 Hex

Features:

  • Only 0-9a-fA-F
  • Length usually even

Example:

666c61677b6865785f656e636f64657d

Python decode:

print(bytes.fromhex("666c61677b6865785f656e636f64657d").decode())

10.3 URL encoding

Example:

%66%6c%61%67%7b%75%72%6c%7d

Python decode:

from urllib.parse import unquote

print(unquote("%66%6c%61%67%7b%75%72%6c%7d"))

10.4 Compressed data signatures

If you see these magic bytes, consider decompression or file recovery:

1f 8b       gzip
78 9c       zlib
50 4b       zip
89 50 4e 47 png
ff d8 ff    jpg

Be sensitive to magic bytes. Sometimes transferred data is a real file, even if extension is fake.

11. A General Troubleshooting Checklist

After receiving a capture, follow this order:

  1. Read challenge description and infer likely protocols.
  2. Use strings to search flag, admin, password, ctf.
  3. In Wireshark, inspect Protocol Hierarchy.
  4. Use Conversations to find high-volume sessions.
  5. Filter common protocols: HTTP, FTP, DNS, ICMP.
  6. For suspicious TCP connections, use Follow TCP Stream.
  7. If HTTP transferred files, use Export Objects.
  8. Try Base64/Hex/URL decode on suspicious strings.
  9. Continue with stego/decompression/script analysis on exported files.
  10. If data is fragmented, extract fields with tshark and merge via script.

Not every challenge requires every step, but this flow prevents blind random clicking.

12. Practical Experience Tips

12.1 Do not only search for flag

Many tasks do not directly expose flag{}. You may first find a password, archive, image, script, or key.

Besides flag, also search:

admin
password
pass
login
upload
download
secret
key
token
ctf

12.2 Prioritize large-data connections

If one TCP connection has significantly more data than others, it often carries key content or files.

12.3 Prioritize plaintext protocols

HTTP, FTP, DNS, ICMP are often the fastest path in CTF traffic problems. If these protocols exist, analyze them first before heavy TLS-focused work.

12.4 Decode abnormal long strings immediately

Unusually long strings in URL, Cookie, POST params, DNS subdomains, or ICMP data are often intentional clues.

12.5 Script repetitive work

If you need to combine hundreds of DNS queries, ICMP payloads, or USB key events, do not do it manually. Extract with tshark, process with Python, and reduce mistakes.

13. Summary

Typical traffic-analysis route in CTF:

Description -> Protocol Statistics -> Conversation Analysis -> Protocol Filtering -> TCP Stream Following -> Data Extraction -> Decoding -> File Analysis

For HTTP: inspect requests, responses, cookies, POST body, and exported files. For FTP: look for credentials and transferred files. For DNS: extract domain labels and attempt concatenation/decoding. For ICMP: inspect data fields. For USB: reconstruct input with HID keyboard mapping.
